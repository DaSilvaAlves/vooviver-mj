## Story

**As a** visitante da landing page da Maria Serqueira,
**I want** ver as secções de apresentação — hero, trust bar, destinos em destaque, comparação de preços, testemunhos e sobre a Maria — com conteúdo visual apelativo e informação clara sobre destinos e preços,
**so that** eu consiga perceber rapidamente a proposta de valor, confiar na consultora e tomar a decisão de pedir um orçamento.

**Depends on:** Story 1.1 (projecto inicializado, Layout.astro, dados JSON, paleta de cores)

## Acceptance Criteria

1. `Hero.astro` implementado com: background de imagem destino (Socotra ou Marrocos — placeholder se asset não disponível), foto da Maria com fallback gracioso se ausente, headline diferenciador ("Do Sahara a Socotra — viagens de grupo com líder e guia desde 870 EUR" ou equivalente), sub-headline com proposta de valor, 2 CTAs ("Pedir Orçamento Gratuito" faz scroll para `#formulario`, "Falar por WhatsApp" abre `https://wa.me/{numero}` — pode ser placeholder).
2. `TrustBar.astro` implementado com: selo RNAVT N.12366, "Centenas de viagens organizadas" (genérico até ter número real), "Resposta em 24h", indicador de métodos de pagamento — visível imediatamente abaixo do hero.
3. `Destinations.astro` implementado: 4 cards de destinos lidos de `src/data/destinations.json`, cada card com imagem placeholder (stock photo ou cor sólida), nome do destino, "desde X EUR/pessoa" (dados reais do JSON), badge "Voos incluídos", botão "Saber Mais" com link `?destino={slug}#formulario` (query string antes do fragment, para não ser descartada pelo browser).
4. Layout dos cards: grid 2x2 em desktop (`md:grid-cols-2`), stack vertical em mobile (1 coluna).
5. `PriceComparison.astro` implementado com tabela comparativa usando dados reais da auditoria: Uganda/Ruanda (VooViver 2.950 EUR vs Pinto Lopes ~9.995 EUR, -70%), Socotra (1.880 EUR, exclusivo PT), Marrocos (870 EUR vs Pinto Lopes ~2.595 EUR, -67%), Tailândia (1.750 EUR vs Pinto Lopes ~4.000 EUR, -56%); nota "Todos os pacotes incluem voo, alojamento e guia".
6. `Testimonials.astro` implementado: secção condicionalmente oculta (`hidden`) quando `src/data/testimonials.json` está vazio; quando tiver dados, mostra 3 cards com texto, nome, destino visitado, data.
7. `AboutMaria.astro` implementado: bio placeholder (3-4 frases genéricas sobre consultora VooViver), foto com fallback gracioso se asset ausente, especialidades (Socotra, Uganda/Ruanda, Marrocos), citação "Porque escolhi ser consultora VooViver" placeholder.
8. Todas as secções implementadas como componentes Astro em `src/components/` e integradas em `src/pages/index.astro` na ordem correcta.
9. Layout completamente responsivo: testado em viewport 375px (iPhone SE) e 1440px (desktop).
10. WCAG 2.1 AA básico: todos os `<img>` têm `alt` descritivo, CTAs têm texto visível, contraste de cor mínimo 4.5:1 para texto normal.
11. `npm run build` executa sem erros após todas as secções integradas.

### Componentes Astro — Padrão de Import

```astro
---
// src/components/Hero.astro
import { Image } from 'astro:assets';
import mariaPhoto from '../assets/maria-serqueira.webp';
---
```

**IMPORTANTE:** `<Image>` do Astro requer que o asset exista em `src/assets/`. Se a foto da Maria não estiver disponível no momento da build, usar uma das seguintes estratégias:
1. Verificar se ficheiro existe antes de importar (não possível em Astro static imports)
2. Usar `<img>` HTML normal com `src` para URL externa (stock photo) como placeholder
3. Usar CSS background-image para o hero

**Recomendação para este story:** usar `<img>` HTML com classes Tailwind para todos os placeholders. Astro `<Image>` será adoptado em Story 1.4 para optimização de performance.

### Acessibilidade — Requisitos Mínimos (NFR-03)

- `<html lang="pt-PT">` — já definido no Layout (Story 1.1)
- Cada secção deve ter `<section>` com `aria-label` ou `<h2>` identificador
- Imagens decorativas: `alt=""`; imagens informativas: `alt` descritivo
- CTAs: texto visível e descritivo (não "Clique aqui")
- Rácio de contraste mínimo: 4.5:1 para texto normal, 3:1 para texto grande
